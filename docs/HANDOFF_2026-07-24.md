## 1. このセッションで追加した機能

### 1-1. 出場校エントリーフォーム
複数人出場する学校のパス発送先・入場証枚数を **1回でまとめて受け付ける** ためのフォーム。

**なぜ作ったか**:
- 従来、選手ごとに事前登録していると、同じ学校のパス発送先が重複入力になり、
  事務局のパス発送作業が学校数に対して二重・三重の手間になっていた。
- 学校単位で1回だけ入力すれば事務局側で発送先を一意管理できる。

**構成**:
- `app/register-school.html` + `js/register-school.js`
- GAS: `actionRegisterSchoolEntry_`, `actionGetSchoolEntryFormConfig_`, `actionCheckSchoolEntryExists_`, `actionAdminGetSchoolEntries_`
- シート: `出場校エントリー`（`SCHOOL_ENTRY_COLS` 参照）
- Drive: `FG出場校エントリー書類/<eventId>/` に承諾書を非公開保存

**仕様**:
- **重複時は上書き**。学校名で照合し、既存行を最新内容で置換。旧承諾書URLは `承諾書履歴` 列に改行区切りで蓄積（Drive上のファイルも履歴として残る）。
- **上書き前に警告**: 学校名入力後に `checkSchoolEntryExists` で照会。既提出があれば連絡事項欄の下に「上書き同意チェックボックス」を表示、未チェックだと送信不可。
- **確認メール**: 新規/更新で件名を出し分け。テンプレは configキー `schoolEntryMailSubject` / `schoolEntryMailSubjectUpdate` / `schoolEntryMailBody` で編集可能。
- **公開期間チェック**: config `schoolEntryFormOpenAt` / `schoolEntryDeadline`。期間外は完全ブロック（フロント・GAS双方で判定）。
- **承諾書テンプレURL**: config `schoolEntryApprovalUrl` で差し替え可能。未設定時は HTML ハードコードのデフォルトURL。

### 1-2. エントリーリスト・受付リスト画面（管理画面）
出走順・受付タグの元データを画面で作成、CSV出力可能に。

**構成**:
- ダッシュボードに「📋 エントリーリスト」「🎫 受付リスト」ナビカード追加
- ページ: `#<eventId>/entry-list` と `#<eventId>/reception`

**エントリーリストページ（2タブ）**:
1. **出走大学リスト**: ▲▼で大学の並び順を編集、女子ペアリング（3ペア）を選択で編集。
2. **エントリーリスト**: FGクラス+女子クラスを大学順で表示、CSV出力。

**受付リストページ（3タブ）**:
1. **選手受付リスト**: 全選手を大学順で（受付タグID付き）
2. **応援学生受付リスト**: 補欠/メカニック/応援学生
3. **出走リスト**: 走行順ソート（A/B/Cドライバー3セクション）

**走行順ロジック**:
```
男子A: 1  〜 N       （大学順）
女子A: N+1 〜 N+M    （女子ペアA担当）
男子B: N+M+1 〜 2N+M （大学順）
女子B: 2N+M+1 〜 2N+2M
男子C: 2N+2M+1 〜 3N+2M
```
N=男子登録大学数 / M=女子ペア数

**設計判断**:
- **studentId の 6文字目から区分を判定**（西暦1桁+大学コード4桁+区分1文字+連番2桁）。
  既存の studentId 体系がそのまま受付タグの ID として使えるため、追加ID発行は不要。
- **大学の並び順と女子ペアリング** は CONFIG シートに JSON 保存（キー: `schoolRunningOrder`, `womenPairings`）。既存の `adminUpdateConfig` が任意キー対応なので新規GASエンドポイントは不要。
- **CSV出力のみ**（PDF/XLSX は未実装）。テンプレは既存 XLSX を貼り付けで運用する想定。

### 1-3. フォーム管理を3タブ化
「フォーム設定 / メール設定 / 書類URL設定」の3タブ構成に整理。

**書類URL設定タブ**（configで一元管理）:
- `docUrlRulebook` — 大会規則書（register-pre / register / register-school がリンク）
- `docUrlPledge` — 誓約書（register-pre / register がリンク）
- `schoolEntryApprovalUrl` — 学校職員の参加承諾書（register-school のアップロード欄からリンク）

**動作**: 設定URLがあれば JS で anchor href を上書き、未設定なら各HTMLハードコードのデフォルトURLをフォールバック。
**設計意図**: 大会ごとに書類が差し替わっても、フォーム再デプロイなしで管理画面から更新可能。かつ「何も設定されていない状態」でリンク切れが起きない安全策。

## 2. 排除・保留した機能とその理由

### 2-1. 出展申込カードの「メール生成ボタン」（実装 → **取り消し**）
- **やろうとしたこと**: カードごとに「✉ メール」ボタンを追加し、宛先・BCC・件名・本文を差し込みでモーダル表示、コピーボタン付き。
- **取り消し理由**: 実装後にユーザーが「詳細モーダルからコピペするのと手間が変わらない」と判断。フィールド数が少なく Outlook 側の操作コストと比べて価値が薄かった。
- **教訓**: 手動運用のコピペ効率化は、フィールド数が多いか繰り返し頻度が高い場合のみ効果あり。少数フィールドなら既存詳細UIで十分。

### 2-2. Outlook 直接操作（提案 → **見送り**）
- **やろうとしたこと**: ブラウザ版 Outlook を`claude-in-chrome`で操作、または PC版 Outlook を`computer-use`で操作して下書き自動作成。
- **見送り理由**: 手動操作より遅く、ログイン状態依存で不安定。かつ社内Outlookへの外部自動操作リスクが不明。ユーザーが「リンク機能があるのかと思った」と発言、期待していたのは MAPI 的な統合であってUI自動操作ではなかった。

### 2-3. GAS 電話・郵便テキスト書式順序反転（Phase 2 として保留）
- **やろうとしたこと**: `setValues` → `setNumberFormat('@')` の順を反転し、書き込み時に先頭ゼロが消えないようにする。
- **保留理由**: 稼働中の GAS 再デプロイに数秒の断が伴うため、稼働落ち着き次第に延期。
- **緩和策**: フロント側で電話・郵便のハイフン入力を許容し、Sheets が数値化しないよう誘導（B緩和として Phase 1 で対応済み）。
- **既失データ**: 先頭ゼロが消えているセルは、シート上で手動修正が必要。

### 2-4. タブコンテンツのフェードインアニメーション（実装 → **廃止**）
- **やろうとしたこと**: タブ切替時に `.tab-content.active` を CSS animation でフェードイン。
- **廃止理由**: `animation-fill-mode` と JS のクラス切替タイミングが噛み合わず、`opacity:0` のまま残る不具合が発生。タブバー自体の横スライドで十分な視覚フィードバックがあるため、コンテンツ側のアニメは断念。
- **現状**: `display:none / block` の即時切替のみ。

### 2-5. 「選手交代フロー」機能（提案 → **未実装**）
- **やろうとしたこと**: C01 → S01 移動、新選手C02 発行のような選手入れ替えを管理画面から一発操作。
- **未実装理由**: ユーザーから「実運用でそこまでの操作は不要」との判断。手動編集で十分。
- **注意**: 過去データの受付リストPDFに「※被入れ替え選手」の注記があるため、運用としては存在するが、機能化の優先度は低い。

### 2-6. エントリー/受付リストのPDF・XLSX直接出力（未実装）
- **やろうとしたこと**: 印刷用CSS or SheetJS で PDF/XLSX に直接エクスポート。
- **未実装理由**: 既存テンプレXLSXにデザインとロゴが仕込まれており、それを崩さずに差し込む方が実用的。CSV出力＋テンプレへ貼り付け運用で十分。
- **将来的な選択肢**: PDFはブラウザ印刷、XLSXは SheetJS で。ただし現状要望なし。

### 2-7. `schoolEntryApprovalUrl` を新キー名に一本化する案（**見送り**）
- **やろうとしたこと**: 書類URL整備の際に `docUrlSchoolApproval` に改名。
- **見送り理由**: 既存の CONFIG シートに保存済みデータのマイグレーションが発生する。稼働中のイベントで運用ミスに直結するため、既存キー名を維持。GAS側でどちらの名前でも動くよう共存。

## 3. 現状の設計判断（巻き戻し禁止）

`README.md` の記述に加えて、このセッションで確立した設計判断:

- **上書き型フォーム**: 出場校エントリーは学校名で重複判定・上書き。事務局でパス発送先の履歴管理はできるが、選手交代など複雑な操作はしない。
- **書類URLは常にフォールバック確保**: HTMLハードコードURL + config上書きの二段構え。configが空でも壊れない。
- **エントリー/受付リスト**: **CSV出力のみ**が正式。テンプレXLSXへの貼り付け運用が前提。
- **走行順のロジック**: 男子A→女子A→男子B→女子B→男子C の並びが正。3-heat以上は想定せず（Cまで）。
